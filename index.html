Three questions the network already knows the answer to

UC1 — SIM swap detection

The bank's fraud system runs forty machine learning models on every wire transfer. None of them ask the one question that matters: did someone steal this customer's phone number twenty minutes ago?

POST /sim-swap/v1/retrieve-dateWhen was the SIM last swapped?Live
POST /sim-swap/v1/checkWas the SIM swapped in the last N hours?Live
UC2 — Number verification

You send an SMS code. They type it back. You call this "verification." The network knows — without sending anything — whether that number belongs to that device. You just can't ask.

POST /number-verification/v1/verifyDoes this number match this device's SIM?Live
UC3 — Location verification

A customer's GPS says Toronto. A ten-dollar app can make any GPS say Toronto. The cell tower the phone is connected to? That's in Bucharest. The network knows. Your fraud system doesn't.

POST /location-verification/v1/verifyIs this device in the claimed area?Live
UC4 — Chained fraud score

SIM swapped. Number doesn't match. GPS spoofed. Any one signal is a red flag. All three together? That's not a customer. That's someone emptying an account.

The developers who couldn't ask

Waterloo

A developer spent three weeks trying to understand the carrier auth flow for SIM swap detection. He gave up and shipped SMS OTP instead. He knew it was less secure. He shipped it anyway because he couldn't figure out how to do better.

Toronto

A fintech wanted to verify that phone numbers actually belonged to the devices submitting loan applications. No test environment existed. They shipped without verification. A fraud ring submitted two hundred fake applications in one weekend.

Vancouver

An e-commerce company wanted to check if a buyer's phone was in the same city as the shipping address. The network knows. Cell towers have been accurate to city-level for a decade. The company couldn't get a test environment. They eat the chargebacks.

The pattern

The infrastructure exists. The standard exists. The capability is running inside the network right now. The last mile — from the carrier's core systems to a developer's laptop — is where everything falls apart. This sandbox closes that gap.

First API call in 90 seconds

No carrier agreement. No approval process. No telco knowledge required.

# 1. Clone and start
git clone https://github.com/KachaJugaad/camara-dev
cd camara-dev && docker compose up

# 2. Make your first CAMARA call (demo key — no sign-up needed)
curl -X POST http://localhost:8080/sim-swap/v1/retrieve-date \
  -H "Authorization: Bearer demo-sandbox-key-rogers" \
  -H "Content-Type: application/json" \
  -d '{"phoneNumber": "+14165550100"}'

# Response: realistic Rogers behavior — ~120ms latency, 4.5% error rate
# {
#   "latestSimChange": null,
#   "_simulation": {"carrier": "Rogers", "latencyMs": 143.2, "simulated": true}
# }
